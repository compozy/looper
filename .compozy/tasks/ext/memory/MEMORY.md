## Shared Decisions
- `internal/core/subprocess` is the shared home for subprocess launch/shutdown plus generic line-framed JSON-RPC transport and initialize helpers; task 01 only rewired ACP onto the extracted process lifecycle primitives, leaving the transport/handshake surface ready for extension-manager tasks.
- Task 03 discovery resolves precedence before enablement filtering and exposes both raw discovered entries and precedence-resolved effective entries. Downstream provider and skill-pack inventories derive from the effective view, so a disabled higher-precedence declaration suppresses lower-level duplicates rather than falling back implicitly.
- Task 05 introduced the shared runtime extension surface under `internal/core/extension`: `Registry`, `RuntimeExtension`, and `ExtensionCaller`. Task 06 should register typed Host API services into `HostAPIRouter`, and task 08 should populate runtime state (`ready` / `draining`) on those same registry entries instead of inventing a second session model.
- Task 06 extends `internal/core/extension` with the concrete Host API business layer: `KernelOps`, `RegisterHostServices`, and the typed request/result structs in `host_helpers.go`. Task 08 should construct real `KernelOps` instances and register them rather than adding a parallel host-service abstraction.
- Task 07 introduces a neutral `model.OpenRunScope` factory plus `model.RunScope` interface so `kernel`, `plan`, and direct `core` adapters can allocate run resources before planning without importing `internal/core/extension` directly. Future extension work should keep the concrete manager/bootstrap logic in `internal/core/extension/runtime.go` and route call sites through the model seam to avoid recreating the `kernel` ↔ `extension` package cycle.
- Task 07 also moves run-scope ownership onto `model.SolvePreparation`; later tasks should use `prep.Journal()`, `prep.EventBus()`, and `prep.RuntimeManager()` instead of carrying separate journal/bus handles, and should close the scope through `prep.CloseJournal(...)` to preserve the manager → journal → bus teardown order.
- Host API memory operations now use exported helpers in `internal/core/memory/store.go` (`ReadDocument`, `WriteDocument`, `ResolveDocumentPath`) so later tasks and SDKs should reuse the same Markdown document semantics instead of writing memory files ad hoc.
- Nested Host API runs propagate ancestry through `model.RuntimeConfig.ParentRunID`; `NewDefaultKernelOps` seeds the parent chain from explicit config or `COMPOZY_PARENT_RUN_ID`, and `host.runs.start` enforces the depth-3 recursion guard centrally.
- `internal/core/run/journal` now exposes `SubmitWithSeq`, and `host.events.publish` uses it to return the actual journal sequence number promised by the Host API contract.
- Task 08 adds the run-scoped extension manager lifecycle around the existing registry entries. `Manager.Start` negotiates initialize over the shared subprocess transport, updates `RuntimeExtension` state to `ready`, and refuses startup on protocol/version/capability/hook contract mismatches instead of degrading silently.
- Task 08 also makes event forwarding a manager concern: extensions with `events.read` receive best-effort `on_event` deliveries through bounded per-extension queues, and queue overflow is surfaced as `extension.event` with payload kind `delivery_dropped`.
- Task 09 completes the command activation matrix via runtime config: `start` and `fix-reviews` always set `EnableExecutableExtensions`, while `exec` keeps the old fast path unless the operator passes `--extensions`.
- Task 09 keeps manager lifecycle in kernel handlers rather than `OpenRunScope`: the kernel starts the manager before `plan.Prepare`, passes the same run scope into scoped `exec`, and preserves cancellation into deferred `RunScope.Close` / `prep.CloseJournal` teardown.
- Task 10 threads generic hook dispatch through the existing run scope by widening `model.RuntimeManager` with `DispatchMutableHook` / `DispatchObserverHook`, then carrying that manager through `run.Execute`, `runshared.Config`, prompt batch params, ACP session requests, and the session-update handler instead of introducing a second extension-specific runtime channel.
- Task 10 also extends the spawned mock extension harness with `COMPOZY_MOCK_APPEND_SUFFIXES_JSON`, which lets future integration tests patch protocol-shaped plan/prompt/agent payloads without adding new fixture binaries.
- Task 11 completes the remaining hook insertion points. `review.pre_fetch`, `review.post_fetch`, and `review.pre_batch` live in `internal/core/plan/prepare.go`; job/run hooks live across `internal/core/run/executor/{runner.go,execution.go}`; and `artifact.pre_write` / `artifact.post_write` are centralized in `defaultKernelOps.writeArtifactFile` so `host.artifacts.write` and `host.tasks.create` share the same extension-aware writer path.
- Task 11 adds stable hook payload helper models in `internal/core/model/hook_types.go` for `FetchConfig`, `FixOutcome`, `JobResult`, and `RunSummary`; later SDK tasks should mirror those shapes rather than redefining ad hoc payload structs.
- Task 12 adds the builtin extension-management CLI under `internal/cli/extension`. `list` renders raw discovered declarations plus an `active` column derived from precedence + enablement, while `inspect`, `enable`, and `disable` resolve by the effective declaration name so operator actions follow ADR-007 precedence.
- Task 12 install/uninstall semantics are intentionally local-first: `install` copies into `~/.compozy/extensions/<name>/` and immediately records disabled user state, while `uninstall` only removes user-scope directories by name and never touches workspace or bundled declarations.
- Task 13 makes declarative assets command-scoped bootstrap data: `start`, `fix-reviews`, `exec`, and `fetch-reviews` now discover enabled extensions once per invocation, activate review-provider and ACP runtime overlays for that command lifetime, and pass the same discovered skill-pack inventory into setup/preflight and `ext doctor`.
- Task 13 keeps declarative review providers as overlay aliases on top of the base registry and ACP adapters as overlay `agent.Spec` entries on top of the built-in catalog; neither the base provider registry nor the bundled skills filesystem is mutated globally.
- Task 14 ships the public Go author SDK under `sdk/extension` plus `sdk/extension/testing`. The SDK duplicates hook payload and Host API structs on the public surface and uses reflection drift tests (`sdk/extension/compat_test.go`) to keep those shapes aligned with runtime internals without importing `internal/` packages.
- Task 14 confirms that `_protocol.md` is normative for the initialize direction: the runtime manager sends `initialize` to the extension subprocess, and both the Go SDK and future SDK/documentation work should model initialize as host-originated even though `task_14.md` contains client-side wording.
- Task 15 ships the TypeScript authoring surface under `sdk/extension-sdk-ts` (`@compozy/extension-sdk`) and `sdk/create-extension` (`@compozy/create-extension`), plus the author docs under `.compozy/docs/extensibility/`. Future extension-authoring work should treat those paths as the canonical public TS entrypoints.
- Task 15 also corrects the public SDK patch shape for `review.post_fetch`: the wire contract and runtime use `issues`, so both SDKs now expose `IssuesPatch` for that hook instead of reusing `EntriesPatch`.

## Shared Learnings
- Host API artifact writes are intentionally scoped to the resolved workspace root or `.compozy/`; both absolute out-of-root paths and `..` traversal collapse into `capability_denied` with `reason=path_out_of_scope`.
- Extension-emitted public events added in task 06 are `task.memory_updated`, `artifact.updated`, and `extension.event`; downstream consumers can rely on those names being documented in `docs/events.md` and available in `pkg/compozy/events`.
- Task 08 adds lifecycle events `extension.loaded`, `extension.ready`, and `extension.failed` with typed payloads under `pkg/compozy/events/kinds`. Later tasks can rely on those names for run startup/shutdown observability instead of inventing new event shapes.
- `internal/core/extension/testdata/mock_extension` is now the canonical spawned test harness for extension-runtime integration tests. Later tasks should extend that binary when they need real subprocess behavior instead of introducing ad hoc fake extensions.
- `internal/core/extension/testdata/sdk_extension` now provides a second spawned fixture that exercises the public Go SDK over real stdio through `internal/core/extension.Manager`; future SDK/doc tasks can reuse it instead of rebuilding another end-to-end author binary.
- Lifecycle RPC audit entries now cover `initialize` and `shutdown`, so later integration tests can assert real extension startup/teardown without needing hook execution to happen first.
- `review.pre_resolve.resolve = false` cleanly skips remote provider resolution while still finalizing local review artifacts, so future tests or SDK examples can avoid live provider dependencies by using that hook response.
- Observer hooks are dispatched before later mutable pipeline steps, but their subprocess completion is asynchronous; recorder-based integration tests should assert partial order or presence for observer hooks instead of assuming strict file-order completion.
- `compozy ext doctor` now reports enabled-extension skill-pack drift plus provider overlay conflicts/overrides from the same declarative inventory used at runtime, so later tasks should extend that surface rather than creating a parallel doctor flow.
- The monorepo `Makefile` now includes `build-extension-sdks` and `publish-extension-sdks`; later release or CI work for npm publishing should reuse those targets instead of adding a parallel packaging path.

## Open Risks
- `_protocol.md` includes `execute_hook.params.hook.name`, but the current manifest `HookDeclaration` model still has no separate declaration-name field. Task 05 uses the canonical hook event string as the effective wire name; if a later task adds manifest-level hook names, `effectiveHookName` must switch to that source.
