# ADR-002: Per-Run Extension Lifetime

## Status

Accepted

## Date

2026-04-10

## Context

Compozy is a per-command CLI, not a daemon. Commands such as `compozy start`, `compozy fix-reviews`, and `compozy exec` spawn a single Go process that runs a discrete invocation and exits. `start` and `fix-reviews` already operate on run artifacts and journals. `exec` also has a run-like execution path, but today it is intentionally optimized as a lighter-weight ad hoc path unless persistence or extra runtime features are requested. The reference extensibility model in agh assumes a long-lived daemon where extensions boot once at startup and stay resident for the life of the daemon, amortizing initialization cost and enabling cross-request state.

Compozy must decide how the extension process model maps onto short-lived, run-scoped invocations. The candidate lifetimes were:
1. Per-run: spawn at the start of each run, tear down at exit.
2. Per-command: spawn for every CLI invocation including trivial read-only commands.
3. Long-lived daemon mode: add a new `compozy daemon` subcommand that hosts extensions across runs.
4. Hybrid: per-run default plus an optional persistent host.

## Decision

**Executable extensions are spawned at the start of each extension-aware Compozy run and shut down when the run ends. One extension subprocess per (run x extension) pair. No daemon mode in v1.**

- `compozy start` and `compozy fix-reviews` always enable executable extensions.
- `compozy exec` enables executable extensions only when the operator passes `--extensions`. Plain `compozy exec` keeps the current lightweight fast path.
- Read-only commands (`compozy tasks list`, `compozy config show`, etc.) do not spawn extension subprocesses.
- Declarative extension assets such as provider registrations and skill-pack resources may still be discovered during command bootstrap without starting the extension subprocess.
- Extensions are initialized before planning begins, not after `plan.Prepare()`, because `plan.*` and `prompt.*` hooks are in scope for v1. Capability negotiation and registry warm-up happen synchronously; any required extension that fails to initialize fails the run by default (configurable per extension).
- On run shutdown (normal, cancelled, or errored), extensions receive a `shutdown` RPC with a deadline. After the deadline, Compozy sends SIGTERM, then SIGKILL, matching the existing ACP subprocess shutdown discipline.

## Alternatives Considered

### Alternative 1: Per-Command Lifetime

- **Description**: Spawn extensions for every Compozy subcommand, including read-only ones.
- **Pros**: Uniform mental model; extensions observe every CLI invocation.
- **Cons**: High startup overhead on trivial commands (e.g., `compozy tasks list` in a shell prompt); wastes time and energy on read-only paths that have no meaningful extension surface.
- **Why rejected**: The interesting surface (events, hooks, jobs) only exists within runs. Loading extensions on read-only commands buys nothing and harms latency.

### Alternative 2: Daemon Mode

- **Description**: Add `compozy daemon` subcommand that stays resident; commands connect to it via a local socket; extensions live in the daemon.
- **Pros**: Cross-run state, cross-run memory, single-process efficiency, warm caches.
- **Cons**: Massive scope increase — requires daemon crash recovery, IPC from the CLI to the daemon, state synchronization, socket management, health checks, and operator docs. Fundamentally changes what Compozy is.
- **Why rejected**: Out of scope for the v1 extensibility feature. Can be added later on top of the per-run model without breaking the extension interface (see "future hybrid" in Implementation Notes).

### Alternative 3: Hybrid (per-run + optional persistent host)

- **Description**: Per-run by default; add an optional `compozy ext serve` long-lived host that extensions can attach to for users who want reuse.
- **Pros**: Flexibility; forward-looking.
- **Cons**: Doubles the test matrix (per-run and persistent paths) and forces every extension author to reason about both lifetimes.
- **Why rejected**: Premature. Per-run is the right starting point; persistent host is a well-defined additive feature that can be added behind the same extension SDK later without breaking changes.

## Consequences

### Positive

- Lifetime maps 1:1 to the journal/run artifact directory, so extension side effects are naturally scoped to a single run's artifacts.
- Crash isolation: an extension crash cannot poison subsequent runs because each run starts with fresh subprocesses.
- Simple mental model for authors: "my process lives for the duration of one run".
- No cross-run state synchronization problem.

### Negative

- Startup cost per run. Measured for ACP today at ~50-200 ms per subprocess spawn (language-dependent). Acceptable at run level where total wall-clock is measured in minutes. `exec` avoids this cost unless `--extensions` is explicitly requested.
- Cross-run caches (e.g., "remember index of all prompts I've rewritten") must be persisted to disk by the extension itself. Compozy provides file-backed memory document APIs for this (ADR-006).

### Risks

- **Risk**: Extensions with expensive initialization (e.g., loading a large embedding model) make every run slower.
  **Mitigation**: Document the lifetime contract clearly; recommend lazy initialization; document the future `compozy ext serve` path as an escape valve if users report pain.
- **Risk**: Per-run lifetime tempts authors to skip proper shutdown handling ("the process will die anyway").
  **Mitigation**: SDK emits warnings when extensions don't handle `shutdown`; enforce in linter/scaffold.

## Implementation Notes

- Extension-aware run scope is constructed before planning begins so `plan.*` and `prompt.*` hooks are possible in v1. The executor still owns in-run teardown, but not the original "create manager after prepare" bootstrap point.
- Each extension subprocess receives the following environment variables set by the manager:
  - `COMPOZY_RUN_ID` — the current run ID
  - `COMPOZY_PARENT_RUN_ID` — propagated from parent when `host.runs.start` recursively launches a run (see ADR-006), enabling loop detection in user-space
  - `COMPOZY_WORKSPACE_ROOT` — absolute path to the resolved workspace root
  - `COMPOZY_EXTENSION_NAME` — own name for logging
- Future hybrid path: a `compozy ext serve` command can be added later. Its process would host the same extension code but expose a local Unix socket; the per-run extension manager would detect the socket and route hook calls through it instead of spawning its own children. No SDK changes required because extensions see the same JSON-RPC messages either way.
