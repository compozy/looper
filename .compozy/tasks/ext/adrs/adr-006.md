# ADR-006: Host API Surface for Extension Callbacks

## Status

Accepted

## Date

2026-04-10

## Context

Extensions receive hook dispatches from the Compozy host but often need to call back to the host to read state, create artifacts, publish events, or launch follow-up work. Without a defined Host API, extensions fall back to direct filesystem writes, environment inspection, and shelling out to `compozy` subcommands — all of which bypass the kernel's audit trail, event bus, and consistency rules.

Three concrete user stories drove this decision:

1. "Inject extra content into the prompt that goes into every execution" — covered by the `prompt.*` mutable hooks.
2. "Run an arbitrary command like `compozy exec ...` at the end of each execution" — requires either shelling out or a host call. Shelling out creates a recursion hazard (the spawned `compozy exec` loads the same extension, triggers the same hook, spawns again).
3. "Create a new task file at the end of a run that captures follow-up work" — needs a way to register an artifact with the kernel so next runs see it, memory indexes update, and the audit log captures the write.

Without a Host API, story 3 forces raw `os.WriteFile`, which breaks the kernel's artifact contract. Story 2 needs a host-managed invocation path that sets `COMPOZY_PARENT_RUN_ID` to break loops. The API also needs to align to Compozy's current state model: task metadata and workflow memory are file-backed, not hidden behind an existing key/value database or a `create-tasks` CLI command.

## Decision

**Compozy ships a minimal Host API in v1 with the following surface. Every method is exposed as a JSON-RPC method callable by the extension over the same stdio transport used for hook dispatch. Every call is capability-checked (ADR-005) and logged to the run's `extensions.jsonl`.**

### v1 Host API

| Namespace | Method | Capability | Purpose |
|---|---|---|---|
| `host.events` | `subscribe(filter)` | `events.read` | Receive lifecycle events from the bus |
| `host.events` | `publish(event)` | `events.publish` | Publish a custom event onto the bus (emits `EventKindExtensionEvent`) |
| `host.tasks`  | `list(workflow)` | `tasks.read` | Enumerate task files in a workflow directory |
| `host.tasks`  | `get(workflow, number)` | `tasks.read` | Read a single task file (frontmatter + body) |
| `host.tasks`  | `create(spec)` | `tasks.create` | Create a new task file; emits `EventKindTaskFileUpdated` |
| `host.runs`   | `start(config)` | `runs.start` | Programmatically start a new run; propagates `COMPOZY_PARENT_RUN_ID` |
| `host.artifacts` | `read(path)` | `artifacts.read` | Read an artifact under `.compozy/` (path validated against allowed roots) |
| `host.artifacts` | `write(path, content, opts)` | `artifacts.write` | Write an artifact under `.compozy/`; emits `EventKindArtifactUpdated` |
| `host.prompts` | `render(templateName, params)` | (none) | Invoke the built-in prompt template renderer (helper, no side effects) |
| `host.memory` | `read(workflow, task_file?)` | `memory.read` | Read a file-backed workflow or task memory document |
| `host.memory` | `write(workflow, task_file?, content)` | `memory.write` | Write a file-backed workflow or task memory document; emits `EventKindTaskMemoryUpdated` |

### Principles

- **Read methods** are side-effect free and return current state synchronously from the host's in-memory/on-disk representation.
- **Write methods** go through the same kernel/service code paths that the CLI uses. Writing a task via `host.tasks.create` emits the same events and updates the same indices as the built-in task-writing service; it must not shell out to a nonexistent `create-tasks` CLI command.
- **Recursive run protection**: `host.runs.start` refuses to start a new run when `COMPOZY_PARENT_RUN_ID` is already set three levels deep (cycle guard). The rule is that nested runs must form an acyclic chain with a bounded depth.
- **Path scoping**: `host.artifacts.read/write` reject any path not under `.compozy/` or the explicit workspace root. No path traversal, no arbitrary filesystem access disguised as an artifact call.
- **No raw SQL, no direct journal manipulation, no direct event-bus internals**. The Host API hides all of those.

## Alternatives Considered

### Alternative 1: No Host API (extensions shell out or write files directly)

- **Description**: Extensions interact with Compozy only through hook dispatches. Any mutation they want to cause happens via `os.WriteFile` or `exec.Command("compozy", ...)`.
- **Pros**: Zero new RPC surface to design, test, or version.
- **Cons**: No audit trail; mutations bypass kernel invariants; recursion hazard requires convention (env-var guards) rather than enforcement; next runs may not see changes because indices weren't updated.
- **Why rejected**: Loses the audit and consistency guarantees that make the extension system safe to trust.

### Alternative 2: Read-Only Host API

- **Description**: Expose `list/get/read` methods but require all mutations to happen via hooks or shellout.
- **Pros**: Smaller surface; limits "what could go wrong" during mutations.
- **Cons**: Story 3 (create follow-up task at end of run) is impossible without the write surface. Forces filesystem cowboys.
- **Why rejected**: Undermines real user stories.

### Alternative 3: Maximal Host API (`host.kernel.dispatch`, raw bus access, direct DB)

- **Description**: Full access to the kernel dispatcher so extensions can invoke any command.
- **Pros**: Maximum power.
- **Cons**: No coherent security model; capability enforcement becomes impossible at the kernel-command granularity; couples extension code to internal types.
- **Why rejected**: Extension API should be a stable, minimal surface. Exposing the kernel dispatcher makes every internal change a breaking change.

## Consequences

### Positive

- Extensions get a clean, documented, versioned mutation surface.
- Every write flows through the same kernel paths as the CLI, guaranteeing consistency.
- The audit trail (`extensions.jsonl`) captures every call, including its arguments and result.
- Recursion protection happens at a single place (`host.runs.start`), not sprinkled across extensions.
- Read methods enable extensions to build rich features without reverse-engineering filesystem layout.
- The memory surface matches the existing Markdown-backed workflow/task memory model instead of inventing a second persistence abstraction.

### Negative

- Every new capability/method requires a small addition to the Host API surface and the SDK type definitions in both Go and TypeScript.
- Extensions cannot access things the host doesn't explicitly expose. For v1 this is a feature, not a bug.

### Risks

- **Risk**: API surface grows without discipline, ending in the same "expose the kernel" anti-pattern.
  **Mitigation**: Require an ADR for every new Host API method or capability added in future releases. Keep method count low and purpose sharp.
- **Risk**: Host API versioning collides with protocol versioning.
  **Mitigation**: Host API is versioned with the protocol handshake (`protocol_version: "1"`). Adding methods is additive; removing methods requires a major bump.

## Implementation Notes

Go SDK shape:

```go
type HostAPI struct {
    Events    *EventsClient
    Tasks     *TasksClient
    Runs      *RunsClient
    Artifacts *ArtifactsClient
    Prompts   *PromptsClient
    Memory    *MemoryClient
}

type TasksClient struct{ rpc *jsonrpc.Client }

func (c *TasksClient) Create(ctx context.Context, spec CreateTaskRequest) (*Task, error) {
    var out Task
    err := c.rpc.Call(ctx, "host.tasks.create", spec, &out)
    return &out, err
}
```

TypeScript SDK shape:

```ts
export class HostAPI {
    constructor(private transport: Transport) {}
    readonly events = new EventsClient(this.transport);
    readonly tasks = new TasksClient(this.transport);
    readonly runs = new RunsClient(this.transport);
    readonly artifacts = new ArtifactsClient(this.transport);
    readonly prompts = new PromptsClient(this.transport);
    readonly memory = new MemoryClient(this.transport);
}
```

Recursion detection:

```go
// In the extension manager, before honoring host.runs.start
parentChain := strings.Split(os.Getenv("COMPOZY_PARENT_RUN_ID"), ",")
if len(parentChain) >= 3 {
    return ErrRecursionDepthExceeded
}
childEnv := append(os.Environ(),
    "COMPOZY_PARENT_RUN_ID="+strings.Join(append(parentChain, currentRunID), ","))
```
