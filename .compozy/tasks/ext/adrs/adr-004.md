# ADR-004: Priority-Ordered Mutation Pipeline for Hooks

## Status

Accepted

## Date

2026-04-10

## Context

Compozy's extension model exposes mutable hooks (ADR-005) at pipeline boundaries where multiple extensions may legitimately want to participate. Example: two extensions both want to decorate a prompt — one to prepend user preferences, one to append workspace context. The host must decide a deterministic execution order and a conflict resolution semantics.

Alternatives considered were: numeric priority pipeline, install-order, named phases (pre/main/post), and mutual exclusion (only one owner per hook).

The ecosystem precedent for extension pipelines that mutate shared data is strong: webpack loaders, vite plugins, rollup plugins, babel plugins, PostCSS plugins, and HTTP middleware all converge on some form of ordered pipeline, usually with explicit priority or explicit ordering declarations.

## Decision

**For every mutable hook, extensions declare a numeric priority in their manifest. Compozy runs them in ascending order. The output of one extension's patch becomes the input for the next. Observers (non-mutating subscribers to the event bus) run in parallel and are not ordered.**

Rules:
- Priority is an integer in `[0, 1000]`. Default 500 when omitted.
- Lower numeric priority runs earlier. Higher numbers run later. Convention: use 100 for "run cheap validators early", 500 for ordinary transforms, 900 for "finalize at the end".
- Ties (same priority across extensions) are broken alphabetically by extension name for determinism.
- Each hook invocation in the chain receives the already-mutated value from the previous extension as its input. Chain of responsibility pattern.
- If any extension in the chain returns an error, the chain aborts and the error propagates to the calling phase. Optional hooks (`required = false`) log and continue; required hooks (`required = true`) fail the run.
- Observers (non-mutating subscribers via `events.Subscribe`) receive the final post-chain value and run asynchronously. They cannot affect chain outcome, and delivery is best-effort under the current bounded event-bus semantics.

## Alternatives Considered

### Alternative 1: Install-Order

- **Description**: Extensions run in the order they were discovered/installed. Filesystem directory listing order, essentially.
- **Pros**: Trivial to implement.
- **Cons**: Not reproducible across machines (filesystem order varies), renaming or reinstalling an extension silently reorders execution, debugging production issues becomes "what's the install order right now".
- **Why rejected**: Determinism is a non-negotiable property of a mutation pipeline.

### Alternative 2: Named Phases (pre/main/post)

- **Description**: Each hook declares a phase: `pre` (validators), `main` (transformers), `post` (finalizers). Within a phase, alphabetical order by extension name.
- **Pros**: Readable phase names convey intent.
- **Cons**: Three phases is often not enough. When the user needs "run between these two main extensions", there's no room. Falls back to phase-cramming or suffix-name hacks (`main-zzz-last`).
- **Why rejected**: Priority numbers are strictly more expressive while preserving the same clarity when authors use well-known values (100, 500, 900).

### Alternative 3: Mutual Exclusion (Single Owner per Mutable Hook)

- **Description**: Only one extension can own a mutable hook. If two declare it, config is invalid and Compozy fails fast.
- **Pros**: Eliminates interaction bugs by construction.
- **Cons**: Rejects legitimate compositions (prefix + suffix prompt decorators, two independent validators). Forces users to merge extensions by hand or choose between them.
- **Why rejected**: Too restrictive; real users want to stack independent transformations.

### Alternative 4: Topological Sort via `before`/`after` Declarations

- **Description**: Extensions declare relative ordering (`before: ["ext-a"], after: ["ext-b"]`); host computes a topological order.
- **Pros**: Expressive when ordering is a true dependency.
- **Cons**: Cycles are possible and painful; authors must know the names of unrelated extensions; detection of impossible orderings happens at runtime.
- **Why rejected**: Higher cognitive load than numeric priorities with no decisive benefit at v1 scale.

## Consequences

### Positive

- Deterministic execution regardless of install order or filesystem layout.
- Author-facing mental model matches familiar ecosystems (webpack, vite, babel).
- Works equally well for 1, 5, or 50 extensions because ordering is explicit.
- Debugging is straightforward: `compozy ext list` shows the effective chain for each hook.

### Negative

- Priority numbers are "magic numbers" without a convention. Compozy must publish a convention doc (100/500/900 buckets).
- Collision handling via alphabetical tiebreak may surprise authors who use identical priorities across extensions. Mitigated by documentation and the ext list CLI showing tiebreak resolution.
- Observer subscribers do not get the same deterministic delivery guarantee as mutable chains. They are observational only.

### Risks

- **Risk**: Two extensions both pick priority 500 and rely on undocumented ordering.
  **Mitigation**: `compozy ext doctor` command warns when multiple extensions share priority on the same hook. Authors can adjust upward.
- **Risk**: An extension panics mid-chain, breaking downstream extensions.
  **Mitigation**: Subprocess isolation — a panicking extension simply closes its JSON-RPC connection. Host catches the closed connection, records an error, and either aborts the run (required hook) or skips the remaining chain entry and continues (optional hook).

## Implementation Notes

Hook dispatch pseudocode in the extension manager:

```go
func (m *Manager) DispatchMutable(
    ctx context.Context,
    hook HookName,
    input any,
) (any, error) {
    chain := m.chainFor(hook) // pre-sorted by priority, then name
    current := input
    for _, ext := range chain {
        patch, err := ext.Call(ctx, "execute_hook", current)
        if err != nil {
            if ext.Required(hook) {
                return nil, fmt.Errorf("hook %s required extension %s failed: %w", hook, ext.Name, err)
            }
            m.logger.Warn("optional hook failed", "ext", ext.Name, "hook", hook, "err", err)
            continue
        }
        current = applyPatch(current, patch)
    }
    return current, nil
}
```

Manifest shape:

```toml
[[hooks]]
event    = "prompt.pre_build"
priority = 200
required = false
timeout  = "2s"
```
