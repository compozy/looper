# ADR-001: Subprocess-Only Extension Model

## Status

Accepted

## Date

2026-04-10

## Context

Compozy needs an extensibility mechanism that lets third parties add lifecycle observers, workflow hooks, providers (IDE/review/model adapters), and skill packs without forking and recompiling the binary. The reference project agh evaluated a three-tier model (L1 Go-native compiled-in, L2 WebAssembly sandboxed, L3 subprocess via JSON-RPC). Compozy must decide whether to adopt the full tier stack, a subset, or a single model for executable extension logic.

Key constraints:
- Compozy is a single-binary, local-first CLI that runs per-command (not a daemon). There is no long-lived host process where an in-process WASM runtime amortizes load cost across many invocations.
- Compozy already spawns subprocesses over JSON-RPC for agent runtimes (Claude Code, Codex, Droid, Cursor) through `internal/core/agent` using the ACP protocol. Subprocess plumbing exists and is well understood.
- Extension authors should be able to work in Go or TypeScript without adding a protobuf or WebAssembly toolchain to their dependency graph.
- YAGNI applies: zero current users, zero measured bottleneck, and no marketplace demand for a sandboxed in-process tier.

## Decision

**Compozy will ship a single subprocess-based execution model (L3 only) for executable extension logic. No L1 Go-native tier and no L2 WebAssembly tier in v1.**

- Every extension feature that executes custom code at runtime runs as an OS subprocess communicating with the Compozy host via JSON-RPC 2.0 over stdio.
- Declarative extension assets such as provider registrations and skill-pack resources may be discovered from manifests without spawning the extension subprocess until their executable path is actually used.
- First-party Compozy code that would qualify as L1 in agh stays as ordinary internal Go code, not exposed through an extension boundary.
- The architecture preserves the option to add a WASM tier later behind the same extension interface if a measured latency problem or a sandbox requirement emerges.

## Alternatives Considered

### Alternative 1: Three-Tier Model (L1 + L2 + L3)

- **Description**: Mirror agh's design with Go-native interfaces for first-party code, Extism/wazero for sandboxed in-process plugins, and subprocess for full-isolation plugins.
- **Pros**: Maximum flexibility; sandboxed hot path for latency-sensitive hooks; broad language support through Wasm PDKs.
- **Cons**: Two additional runtimes to implement, test, document, and operate; ~5-8 MiB binary growth from wazero; zero current demand; duplicates Go-native and subprocess surfaces without a proven bottleneck.
- **Why rejected**: YAGNI — no measured hook latency budget exists yet, no user base is asking for a sandbox, and Compozy's per-run CLI model means WASM startup would happen every run, which erases most of its amortization benefit.

### Alternative 2: L1 Go-Native Only (Compile-Time Plugins)

- **Description**: Expose a Go interface and let users fork, implement it, and rebuild the binary.
- **Pros**: Zero IPC overhead; compile-time type safety.
- **Cons**: Fork-to-extend model has very high friction; blocks non-Go authors entirely; forces extension distribution to rebuild every Compozy release; no hot install.
- **Why rejected**: Does not satisfy the core requirement of extending Compozy without recompiling.

### Alternative 3: HashiCorp go-plugin (gRPC Subprocess)

- **Description**: Subprocess model using gRPC + protobuf instead of JSON-RPC.
- **Pros**: Strong typing on the wire; battle-tested in Terraform, Vault, Packer.
- **Cons**: Adds protobuf toolchain as a barrier for TypeScript and other-language authors; heavier runtime; duplicates concepts with the existing JSON-RPC-based ACP machinery in `internal/core/agent`.
- **Why rejected**: JSON-RPC over stdio is already the lingua franca of the Compozy process boundary (ACP) and of the broader agent ecosystem (LSP, MCP). Consistency wins.

## Consequences

### Positive

- One protocol, one executable lifecycle, one debugging surface for authors to learn.
- Reuses the subprocess plumbing already proven in `internal/core/agent`.
- Supports Go, TypeScript, Python, or any language with a JSON-RPC library.
- Keeps the binary lean (no embedded Wasm runtime).
- Matches per-run CLI lifetime cleanly — no "daemon boot amortization" story required.

### Negative

- Every hook invocation pays IPC cost (~100-500 µs per call on local stdio). Acceptable for workflow-level hooks that fire on run/job/prompt boundaries but would be measurable on a per-tool-call or per-message tight loop.
- Compozy cannot offer a "safe in-process extension" tier for untrusted code. Not a concern for local-first tool but limits a future marketplace.

### Risks

- **Risk**: A future latency-sensitive hook category (e.g., per-stream-chunk filter) may become painful on subprocess IPC.
  **Mitigation**: Keep hook names and payload shapes stable so a future WASM tier can implement the same interface transparently.
- **Risk**: Extension authors may expect a sandbox and be surprised that their extension can read arbitrary files under the host user.
  **Mitigation**: Document the trust model clearly; capability-based security (ADR-005) at the Host API layer; extensions must be installed explicitly per workspace or user.

## Implementation Notes

- All executable extension code lives under `internal/core/extension/` and `sdk/extension/` (Go SDK) plus `@compozy/extension-sdk` (TypeScript SDK published to npm).
- Manifests may describe non-executable assets (`[[providers.*]]`, `[resources.skills]`) that are loaded without changing this subprocess-only decision, because the decision is about how third-party code executes, not how metadata is discovered.
- The extension interface is deliberately tier-agnostic — no "Tier" enum is exposed — so a future WASM implementation can slot in without API changes.
- ACP subprocess logic is extracted into `internal/core/subprocess/` (see ADR-003) and consumed by both agent and extension managers.
