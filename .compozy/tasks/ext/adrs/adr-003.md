# ADR-003: JSON-RPC 2.0 over stdio with Shared `internal/core/subprocess` Package

## Status

Accepted

## Date

2026-04-10

## Context

Extensions need a well-defined IPC protocol to talk to the Compozy host. The decision covers two tightly coupled choices: which protocol to speak on the wire, and whether to create a new subprocess plumbing package or reuse and generalize the existing ACP subprocess code in `internal/core/agent`.

Compozy already runs agent runtimes (Claude Code, Codex, Droid, Cursor) as subprocesses that speak JSON-RPC 2.0 over stdio via the ACP SDK. The spawn, handshake, signal handling, health-check, and graceful shutdown logic is non-trivial and platform-specific (Unix process groups via `Setpgid: true` in `process_unix.go`, Windows-native process termination in `process_windows.go`).

Forking that logic into a second implementation for extensions would duplicate tricky code, invite drift, and double the test matrix.

## Decision

**Extensions communicate with the Compozy host via JSON-RPC 2.0 over stdin/stdout using a newly extracted shared package `internal/core/subprocess`. The existing ACP code in `internal/core/agent` and the new extension code in `internal/core/extension` both import this package.**

- Framing: one JSON object per line, UTF-8, `\n` delimited. Blank lines ignored. No batch requests in v1. This deliberately follows Compozy's current ACP subprocess implementation rather than LSP's `Content-Length` framing.
- Both directions: the host can call the extension (hook dispatch) and the extension can call back into the host (Host API). IDs correlate request/response pairs; responses may arrive out of order.
- Protocol version: exact-match string `"1"` in the initialize handshake. Unknown future versions return `-32602 invalid params`.
- Shared package contents: `Process` struct (spawn, wait, kill), `Transport` (line framing, encode/decode), `Handshake` (initialize negotiation), `Signals` (graceful shutdown with SIGTERM → SIGKILL escalation), platform files for Unix and Windows.
- ACP-specific logic (ACP method names, ACP capability strings) remains in `internal/core/agent`. Extension-specific logic (extension method names, capability list, hook routing) lives in `internal/core/extension`. The shared package is protocol-agnostic except for the JSON-RPC envelope.

## Alternatives Considered

### Alternative 1: New Independent Package, No Refactor

- **Description**: Write a second subprocess implementation in `internal/core/extension/transport` from scratch.
- **Pros**: Zero refactor risk to the existing ACP code path.
- **Cons**: Duplicates ~1500 LOC of spawn/handshake/shutdown logic; guaranteed drift as one side gets bug fixes the other does not; doubles the platform-specific test surface.
- **Why rejected**: The refactor is straightforward and the duplication cost is permanent.

### Alternative 2: Unix Domain Sockets

- **Description**: Each extension listens on `/tmp/compozy-ext-<run-id>-<name>.sock`; host connects as a client.
- **Pros**: Multi-client possible; external tooling like `nc`/`socat` can attach to the socket for debugging.
- **Cons**: Socket file management (cleanup on crash, permission bits, name collisions), platform parity problem (Windows needs TCP fallback or named pipes), no value over stdio for single-host/single-client communication.
- **Why rejected**: Solves a problem we do not have and introduces cleanup and portability burdens.

### Alternative 3: gRPC with Protobuf

- **Description**: Use HashiCorp go-plugin patterns with gRPC.
- **Pros**: Strong typing on the wire, proven ecosystem.
- **Cons**: Protobuf toolchain as a dependency for every extension author, especially painful for TypeScript. Heavier runtime footprint. Diverges from the JSON-RPC pattern already in use.
- **Why rejected**: See ADR-001. Consistency with ACP and the broader agent ecosystem (LSP, MCP) wins.

## Consequences

### Positive

- Single source of truth for subprocess lifetime, health, and shutdown.
- Bug fixes to process management help both ACP and extensions.
- Ecosystem familiarity: any developer who has written an LSP or MCP server recognizes the protocol shape.
- TypeScript authors can use a plain `JSON.parse`/`JSON.stringify` with a stdin reader and need no extra tooling.

### Negative

- The refactor touches a working, tested path (`internal/core/agent`). Must be done carefully with the existing ACP test suite as the safety net.
- The shared package must stay protocol-agnostic enough to serve both ACP's fixed method set and extension's dynamic capability list. Small risk of abstraction drift.

### Risks

- **Risk**: The extracted package forces an awkward shape that neither caller enjoys.
  **Mitigation**: Extract in two passes — first move files verbatim into the new package, then introduce small surface-level abstractions only when both callers actually need them.
- **Risk**: Line-delimited JSON runs into problems with unusually large payloads (>10 MiB).
  **Mitigation**: Set a 10 MiB max message size; return `-32603 internal error` with `data.reason = "message_too_large"` when exceeded; document the limit in the protocol spec.

## Implementation Notes

Proposed package tree:

```
internal/core/subprocess/
    process.go         # Launch, Wait, Kill, Env management
    transport.go       # JSON-RPC envelope framing, encode/decode, line reader
    handshake.go       # Initialize/Initialized exchange, version negotiation
    signals.go         # SIGTERM/SIGKILL escalation, deadline handling
    process_unix.go    # Setpgid, process group kill (moved from internal/core/agent)
    process_windows.go # Windows-native termination (moved from internal/core/agent)
    subprocess_test.go
```

Callers:
- `internal/core/agent/client.go` imports `subprocess.Launch(...)` and layers ACP-specific initialize over the returned transport.
- `internal/core/extension/manager.go` imports `subprocess.Launch(...)` and layers the extension initialize handshake over the returned transport.

Migration order: move files, update agent imports, run `make verify`, add extension consumer, add tests. All done behind the existing ACP integration tests as the regression safety net.
