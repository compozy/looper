# ADR-007: Three-Level Discovery with TOML-First Manifest

## Status

Accepted

## Date

2026-04-10

## Context

Extensions must be discovered during command bootstrap. Some commands only need declarative assets such as provider overlays or skill-pack metadata, while run-scoped commands also need executable extension processes. Compozy already has an established config convention in `.compozy/config.toml` (workspace-scoped TOML) and bundles assets via `go:embed` in `skills/embed.go`. The decision covers two related choices: where extensions live on disk, and what format their manifest files use.

Three discovery location strategies were considered:
1. Workspace-only — extensions live in `.compozy/extensions/` inside the project.
2. User-only — extensions live in `~/.compozy/extensions/`, shared across projects.
3. Three-level — bundled (embedded in binary) + user (`~/.compozy/extensions/`) + workspace (`.compozy/extensions/`).

For manifest format, the options were TOML (matching Compozy's existing `.compozy/config.toml`), JSON (friendlier for TypeScript authors shipping alongside `package.json`), or both.

## Decision

**Extensions are discovered from three levels in precedence order: workspace > user > bundled. Each level has its own directory. The manifest is TOML-first (`extension.toml`), with JSON fallback (`extension.json`) for authors who prefer it.**

### Discovery locations

| Level | Location | Purpose | Writable by |
|---|---|---|---|
| **Bundled** | Embedded via `go:embed` under `internal/core/extension/builtin/` | First-party extensions shipped with Compozy | Compozy release process |
| **User** | `~/.compozy/extensions/<name>/` | Personal extensions shared across the user's projects | User manually or via `compozy ext install` |
| **Workspace** | `<workspace-root>/.compozy/extensions/<name>/` | Project-scoped extensions, committable to git | Anyone with repo write access |

### Precedence

When two levels declare an extension with the same name, **workspace wins over user wins over bundled**. This lets a workspace pin a specific version of an extension or temporarily override a bundled/user version for experimentation, without editing global state.

### Enablement

- Discovery and enablement are separate concerns.
- Bundled extensions default enabled.
- User and workspace extensions default disabled until the local operator enables them.
- Enablement state for workspace extensions is stored outside the repo so cloning a workspace does not automatically activate its extension code on another machine.
- Declarative assets such as provider registrations and skill-pack resources follow the same enablement rule: if the extension is disabled, its overlays are not applied.

### Manifest loading

- Loader first tries `<extension-dir>/extension.toml`. If present, parses and returns.
- If absent, tries `<extension-dir>/extension.json`. If present, parses and returns.
- If both absent, the directory is ignored with a warning.
- If both present, `extension.toml` wins and a warning is logged about the unused JSON file.

### Extension directory layout

```
extensions/
    my-ext/
        extension.toml           # or extension.json
        bin/
            my-ext               # subprocess entry (if any)
        skills/                  # optional resources declared in manifest
            my-skill/
                SKILL.md
        resources/               # optional data files
```

## Alternatives Considered

### Alternative 1: Workspace-Only

- **Description**: Extensions only exist in `.compozy/extensions/`. No bundled, no user.
- **Pros**: Zero global state; reproducible because extensions travel with the repo; every project gets the exact extensions its users committed.
- **Cons**: Forces duplication when a developer wants the same extension across many projects; no way to ship first-party extensions with Compozy releases.
- **Why rejected**: Bundled level is valuable for first-party extensions and user level solves the personal-preference case. Losing both for marginal simplicity is a poor trade.

### Alternative 2: User-Only

- **Description**: Extensions only exist in `~/.compozy/extensions/`.
- **Pros**: Install once, use everywhere.
- **Cons**: Breaks CI reproducibility — the CI machine needs the same extensions installed or behavior differs from local runs; no project-specific configuration; no per-repo version pinning.
- **Why rejected**: Reproducibility across local and CI is a core Compozy value.

### Alternative 3: Workspace + User, No Bundled

- **Description**: Drop bundled. Ship extensions as separate installable packages.
- **Pros**: Smaller binary.
- **Cons**: First-party extensions become a distribution problem; `compozy setup` has to fetch them from somewhere; versioning drifts from the main binary.
- **Why rejected**: Having first-party extensions compiled in means they always match the binary version. Size cost is acceptable.

### Alternative 4: JSON-Only Manifest

- **Description**: Require `extension.json` only.
- **Pros**: Natural for TypeScript authors; one format to parse.
- **Cons**: JSON has no comments natively and no multi-line strings; diverges from Compozy's existing TOML convention; human edits are painful.
- **Why rejected**: TOML is the house style and is friendlier for the most common case of humans writing manifests.

## Consequences

### Positive

- Clear precedence avoids ambiguity when users want to override.
- Bundled level enables Compozy to ship first-party extensions without a distribution story.
- Workspace level enables version pinning and reproducibility across machines and CI.
- User level satisfies personal-preference use cases.
- Separating discovery from enablement prevents repo-shipped extensions from auto-executing on clone.
- TOML default aligns with existing `.compozy/config.toml` and reduces author friction.
- JSON fallback accommodates TypeScript-first projects where `extension.json` can live next to `package.json` and be generated by tooling.

### Negative

- Two manifest parsers to maintain. Mitigated by using standard libraries and a shared validation layer after parse.
- Precedence rules must be documented clearly; novices may wonder why `compozy ext list` shows a user extension when a workspace one exists with the same name.

### Risks

- **Risk**: Name collisions between levels confuse users.
  **Mitigation**: `compozy ext list` prints the effective level in a column; `compozy ext inspect <name>` shows all declarations and explains which one won and why.
- **Risk**: A user bundles a skill pack in the workspace that conflicts with a bundled skill of the same slug.
  **Mitigation**: Same precedence rule (workspace wins); warning logged when overriding a bundled skill.

## Implementation Notes

Discovery order during command bootstrap:

```go
func (m *Manager) Discover(ctx context.Context) ([]Manifest, error) {
    seen := make(map[string]Manifest)

    // Bundled (lowest precedence)
    for _, m := range loadBundled() { seen[m.Name] = m }

    // User
    if home, err := os.UserHomeDir(); err == nil {
        for _, m := range loadDir(filepath.Join(home, ".compozy", "extensions")) {
            seen[m.Name] = m
        }
    }

    // Workspace (highest precedence)
    for _, m := range loadDir(filepath.Join(workspaceRoot, ".compozy", "extensions")) {
        seen[m.Name] = m
    }

    return values(seen), nil
}
```

Both parsers share a post-parse validation pass that checks: required fields present, capability names known, `min_compozy_version` satisfied, hook names recognized, priority in range.

Bundled extensions are discovered via `embed.FS` at compile time. They live under `internal/core/extension/builtin/<name>/` and are mounted into the discovery pipeline as a synthetic filesystem.
