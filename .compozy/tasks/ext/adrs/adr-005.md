# ADR-005: Capability-Based Security Without Trust Tiers

## Status

Accepted

## Date

2026-04-10

## Context

Extensions can read sensitive workspace data (PRDs, TechSpecs, tokens in env), mutate prompts that reach paid LLM providers, write files under `.compozy/`, and spawn arbitrary subprocesses. Compozy needs a security model that communicates risk to users, limits blast radius, and provides an audit trail without adding friction that kills the feature.

agh's model adds source-trust tiers â€” `bundled`, `user`, `workspace` (all trusted) and `marketplace` (restricted). That model exists because agh envisions a public marketplace where untrusted code reaches end users.

Compozy's context is different:
- Compozy is a local developer tool. There is no marketplace and no plan to have one.
- Workspace extensions are committed to git and reviewed like any other workspace code.
- User-scoped extensions are installed explicitly by the user, who is the sole accountable party.
- Bundled extensions are first-party code shipped in the binary.
- The realistic threat model is "extension does something surprising the author didn't intend", not "attacker publishes hostile extension to public registry".

## Decision

**Compozy adopts declarative capabilities without source-trust tiers.**

- Every extension lists the capabilities it needs in its manifest `[security]` section (e.g., `events.read`, `prompt.mutate`, `artifacts.read`, `artifacts.write`, `runs.start`, `tasks.create`, `memory.read`, `memory.write`, `subprocess.spawn`, `network.egress`).
- On install, `compozy ext install <path>` prints the requested capability list to the user and requires explicit confirmation (`--yes` flag for non-interactive/CI).
- Bundled extensions default enabled. User and workspace extensions default disabled until the local operator explicitly enables them. This avoids auto-activating repo-shipped workspace extensions on clone.
- The host enforces capabilities at every Host API call and every hook dispatch. An extension that calls `host.tasks.create(...)` without `tasks.create` capability receives error `-32001 capability_denied` with structured data listing the missing grant.
- There is no automatic elevation, no "marketplace" restriction tier, and no dynamic capability request at runtime. What's in the manifest at install time is what the extension ever gets.
- Every capability exercised in a run is logged to the audit trail in `.compozy/runs/<run-id>/extensions.jsonl` so users can retroactively inspect what an extension touched.

## Alternatives Considered

### Alternative 1: Capabilities + Source-Trust Tiers (Full agh model)

- **Description**: Adopt agh's full bundled/user/workspace/marketplace tier system with per-tier capability allowlists.
- **Pros**: Defense in depth; future-proof for a marketplace.
- **Cons**: Overkill for local-first tool with no marketplace; user confusion ("why is workspace trusted but marketplace not?"); extra code to maintain.
- **Why rejected**: Compozy has no marketplace and no immediate plan for one. Adding the tier infrastructure now means carrying complexity that only pays off if a marketplace appears.

### Alternative 2: Process Isolation Only (No Capabilities)

- **Description**: The only security boundary is the OS process boundary. No capability checks at Host API or hook dispatch.
- **Pros**: Simplest to implement.
- **Cons**: No audit trail; cannot limit an observer extension from mutating artifacts; extensions can accidentally (or intentionally) drift from their declared intent.
- **Why rejected**: Loses the audit property that makes "what did this extension actually do?" answerable.

### Alternative 3: Capabilities + Interactive Consent on First Use

- **Description**: Declarative capabilities plus Compozy interactively prompts the user the first time an extension exercises a capability ("ext-foo wants to write task files â€” allow?"). Records consent in `.compozy/extensions.lock`.
- **Pros**: Explicit per-capability consent; users aware of behavior as it happens.
- **Cons**: Breaks non-interactive/CI use, the main habitat of Compozy. Adds friction even with `--yes` flag because fresh installs would trip every run.
- **Why rejected**: Hostile to automation. Install-time confirmation with a clear capability list achieves the same awareness without breaking CI.

## Consequences

### Positive

- One security surface to understand: the install prompt and the capability list.
- Workspace extensions are visible in the repo but not active until explicitly enabled by the local operator.
- Audit trail (`extensions.jsonl`) exists without friction.
- Extensions are forced to declare intent in the manifest, which is checkable in code review for workspace extensions.
- Extensions that exceed their declared capabilities fail fast with a clear error â€” "you asked for `events.read` but called `host.tasks.create`".

### Negative

- Coarse-grained capabilities are not fine-grained. An extension with `artifacts.write` can write to *any* artifact path, not just a subtree. Future work may introduce path scopes if needed.
- Users may click through the install prompt or enable a workspace extension without reading. Mitigated by printing the capability list in color and requiring typed confirmation (not just Enter) for high-risk capabilities like `network.egress`.
- This is not an OS sandbox. Advisory capabilities such as `network.egress` and `subprocess.spawn` declare intent and improve auditability, but they do not hard-block behavior at the operating-system boundary.

### Risks

- **Risk**: Users install an extension that requests excessive capabilities ("requires all of them").
  **Mitigation**: `compozy ext doctor` flags extensions whose declared capabilities exceed observed usage after N runs, suggesting the user reinstall with a stricter manifest.
- **Risk**: A capability list grows over time and old extensions become stale.
  **Mitigation**: Capability taxonomy is versioned with the protocol version; new capabilities don't break old extensions.

## Implementation Notes

Initial capability taxonomy (v1):

| Capability | Grants |
|---|---|
| `events.read` | Subscribe to lifecycle events from the bus |
| `events.publish` | Publish custom events via `host.events.publish` |
| `prompt.mutate` | Register mutable `prompt.*` hooks |
| `plan.mutate` | Register mutable `plan.*` hooks |
| `agent.mutate` | Register mutable `agent.*` hooks |
| `job.mutate` | Register mutable `job.*` hooks |
| `run.mutate` | Register mutable `run.*` hooks |
| `review.mutate` | Register mutable `review.*` hooks |
| `artifacts.read` | Call `host.artifacts.read` |
| `artifacts.write` | Call `host.artifacts.write` and register `artifact.*` hooks |
| `tasks.read` | Call `host.tasks.list`, `host.tasks.get` |
| `tasks.create` | Call `host.tasks.create` |
| `runs.start` | Call `host.runs.start` (with loop detection via `COMPOZY_PARENT_RUN_ID`) |
| `memory.read` | Call `host.memory.read` |
| `memory.write` | Call `host.memory.write` |
| `providers.register` | Declare entries under `[[providers.*]]` |
| `skills.ship` | Declare `[resources] skills = [...]` |
| `subprocess.spawn` | Spawn OS child processes from within the extension body (advisory â€” OS cannot block this) |
| `network.egress` | Make outbound network calls (advisory â€” OS cannot block this either) |

Capabilities `subprocess.spawn` and `network.egress` are advisory â€” the OS does not prevent a subprocess from doing these things. They exist for declarative intent and audit, not enforcement. Document this clearly.

Audit entries in `.compozy/runs/<run-id>/extensions.jsonl`:

```json
{"ts": "2026-04-10T14:15:22.123Z", "extension": "compozy-ext-userprefs", "method": "host.artifacts.write", "capability": "artifacts.write", "target": ".compozy/tasks/feat-x/task_05.md", "result": "ok"}
```
